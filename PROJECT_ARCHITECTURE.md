# 🎓 Ahmad Foundation Preparation Portal — Architecture & API Documentation

This document provides a highly detailed, comprehensive overview of the **Ahmad Foundation MCQ Test & Prep Portal** project architecture. It describes what the project does, the roles of the frontend and backend layers, how Mongoose models are structured and interrelated, and provides an exhaustive route-by-route API reference.

---

## 📋 1. Project Overview

The **Ahmad Foundation MCQ Test Portal** is a full-stack, secure online preparation and practicing web application primarily built for grade-level tuition students (e.g., matriculation 9th and 10th grades). The goal of the platform is to allow students to select classes/subjects, unlock educational materials (notes, timed mock exams), practice multiple-choice questions, and track their test scores in real-time.

### Core Value Proposition
- **Flexible Monetization/Gating**: Access to classes and notes can be gated. Students unlock courses via **coupon codes** (generated by admins) or by submitting a **payment request** (easypaisa, jazzcash, whatsapp chat approval).
- **Anti-Cheat Measures**: Scores are calculated 100% server-side. The exam interface disables right-click, copy-paste, and DevTools keyboard shortcuts, and verifies duration using absolute time differences.
- **Hierarchical Layouts**: Dynamic, cascading data paths (Class ➔ Subject ➔ Test series ➔ MCQs / Notes).
- **Comprehensive Administration**: Admins can seed and manage students, classes, subjects, tests, MCQs (single or bulk JSON uploads), generate coupons, upload prep PDFs, and review student grades.

## 🛠️ 2. Frontend vs. Backend Roles

### 🖥️ The Frontend Role
The frontend is built with **React 19**, **Next.js 15 (App Router)**, and **Tailwind CSS 4**. It is responsible for the user interface, routing, layout animations, client-side session validation, and exam controls.

Key features handled by the Frontend include:
1. **Interactive Dashboard**: Dynamically polls approved classes, subjects, and test series. Unapproved classes prompt payment request or coupon redemption modals.
2. **Live Timed Exam Portal**:
   - Manages a countdown timer matching exam duration.
   - Prevents cheats through direct event handlers (`onCopy`, `onCut`, `onPaste`, `onContextMenu` are completely blocked).
   - Tracks current question selections, supporting a clean paginated or single-screen viewport.
3. **Admin Dashboard Interface**:
   - Multi-tab management system (using `AdminShell.tsx`) allowing tables to perform CRUD operations on classes, subjects, tests, and MCQs.
   - Handles file uploads (such as PDF notes and image/video gallery assets).
   - Supports bulk JSON importing of MCQs.

### ⚙️ The Backend Role & Security
The backend is powered by **Next.js API Routes** running on Node.js, and integrates with **MongoDB via Mongoose**. It handles all critical logical flows, authorization, secure storage, rate limiting, and quiz grading.

Key features handled by the Backend include:
1. **Custom PBKDF2 Session Authentication**:
   - Avoids third-party OAuth, utilizing standard Node `crypto` algorithms.
   - Passwords are salted and hashed via PBKDF2 (1000 iterations, SHA-512).
   - Session tokens are signed via **HMAC-SHA256** and packed in Base64URL encoding as HTTP-only secure cookies (`auth-token`).
2. **Server-Side MCQ Scoring (Anti-Cheat)**:
   - When a student completes an exam, the frontend submits *only* the question IDs and chosen options.
   - The backend pulls the matching active MCQs from the database, compares option indexes, computes the score, percentage, and saves the attempt. The correct answers are never sent to the client during an exam.
3. **Sequential Test Unlocking**:
   - Guarantees that a student must complete "Test 1" (e.g., scoring in it or completing it) before "Test 2" under the same subject becomes available.
4. **Rate Limiting**:
   - Employs an IP-based token-bucket/rate-limit check (e.g., maximum of 3 test submissions per minute or 5 login attempts per minute).

## 🗄️ 3. Database Model Architecture

The database consists of **16 Mongoose models** establishing strong relational bonds via ObjectIds.

```mermaid
erDiagram
    User ||--o{ ClassAccess : requests
    User ||--o{ TestSubmission : submits
    Class ||--o{ Subject : contains
    Class ||--o{ ClassAccess : targets
    Class ||--o{ Coupon : unlocks
    Subject ||--o{ Test : contains
    Subject ||--o{ Note : contains
    Test ||--o{ MCQ : contains
    Test ||--o{ TestSubmission : evaluated
    Note ||--o{ NoteAccess : records
    NoteCoupon }|--|| Class : targets
```

### Detailed Schemas and Indexes

#### 1. `User`
- **Fields**: `name` (String), `email` (String, unique, lowercase), `passwordHash` (String), `salt` (String), `isDeleted` (Boolean), `deletedAt` (Date).
- **Indexes**: Unique filter index on `email` where `isDeleted: false`.

#### 2. `Admin`
- **Fields**: `email` (String), `password` (String, plaintext comparison against environment seeded value).

#### 3. `Class`
- **Fields**: `name` (String, unique), `isDeleted` (Boolean), `deletedAt` (Date).
- **Indexes**: Unique index on `name` where `isDeleted: false`.

#### 4. `Subject`
- **Fields**: `name` (String), `classId` (Ref Class), `isDeleted` (Boolean), `deletedAt` (Date).
- **Indexes**: Relational index on `classId`, and unique compound index `{ name: 1, classId: 1 }` where `isDeleted: false`.

#### 5. `Test`
- **Fields**: `name` (String), `classId` (Ref Class), `subjectId` (Ref Subject), `isDeleted` (Boolean), `deletedAt` (Date).
- **Indexes**: Relational indexes on `classId` and `subjectId`. Compound unique index `{ name: 1, subjectId: 1 }` where `isDeleted: false`.

#### 6. `MCQ`
- **Fields**: `question` (String), `options` (Array of exactly 4 non-empty strings), `correctAnswer` (Number, 0-3), `classId` (Ref Class), `subjectId` (Ref Subject), `testId` (Ref Test), `isDeleted` (Boolean), `deletedAt` (Date).
- **Indexes**: Relational indexes on `classId`, `subjectId`, and `testId`. Compound unique index `{ question: 1, testId: 1 }` where `isDeleted: false` to avoid double entries.

#### 7. `TestSubmission`
- **Fields**: `userId` (Ref User), `classId` (Ref Class), `subjectId` (Ref Subject), `testId` (Ref Test), `totalQuestions` (Number), `attemptedCount` (Number), `score` (Number), `percentage` (Number), `startTime` (Date), `submittedAt` (Date), `durationSeconds` (Number), `answers` (Array of `{ mcqId: ObjectId, selectedOption: Number }`).
- **Indexes**: Relational indexes on `userId`, `classId`, `subjectId`, and `testId`.

#### 8. `ClassAccess`
- **Fields**: `userId` (Ref User), `classId` (Ref Class), `status` ("pending", "approved", "rejected"), `paymentMethod` ("easypaisa", "jazzcash", "whatsapp", "coupon"), `paymentDetails` (String).
- **Indexes**: Unique compound index `{ userId: 1, classId: 1 }` to prevent redundant access requests.

#### 9. `Coupon`
- **Fields**: `classId` (Ref Class), `hashedCoupon` (String, bcrypt hash of coupon code), `isUsed` (Boolean), `usedBy` (Ref User), `usedAt` (Date), `isActive` (Boolean).
- **Methods**: `compare(raw: string)` matches raw text against `hashedCoupon` via `bcrypt.compare`.

#### 10. `Note`
- **Fields**: `name` (String), `description` (String), `imageUrl` (String), `pdfUrl` (String), `classId` (Ref Class), `subjectId` (Ref Subject), `requiresCoupon` (Boolean), `isDeleted` (Boolean), `deletedAt` (Date).
- **Indexes**: Compound unique index `{ name: 1, classId: 1, subjectId: 1 }` where `isDeleted: false`.

#### 11. `NoteAccess`
- **Fields**: `userId` (Ref User), `noteId` (Ref Note), `classId` (Ref Class), `subjectId` (Ref Subject), `unlockedVia` ("coupon", "class_access"), `unlockedAt` (Date).
- **Indexes**: Unique compound index `{ userId: 1, noteId: 1 }`.

#### 12. `NoteClassAccess`
- **Fields**: `userId` (Ref User), `classId` (Ref Class), `status` ("pending", "approved", "rejected"), `paymentMethod` ("easypaisa", "jazzcash", "whatsapp", "coupon"), `paymentDetails` (String).
- **Indexes**: Unique index on `{ userId: 1, classId: 1 }`.

#### 13. `NoteCoupon`
- **Fields**: `classId` (Ref Class), `subjectId` (Ref Subject, optional), `hashedCoupon` (String, bcrypt hash), `isUsed` (Boolean), `usedBy` (Ref User), `usedAt` (Date), `isActive` (Boolean).

#### 14. `Course`
- **Fields**: `name` (String, unique).

#### 15. `GalleryAsset`
- **Fields**: `src` (String), `title` (String), `alt` (String), `category` (String), `type` ("image", "video"), `featuredOnHome` (Boolean), `orderOnHome` (Number).

## 📡 4. Comprehensive API Route Documentation

This section details all **44 API Routes** in the portal, grouped logically by domain.

---

### Group A: Student Authentication Routes (`app/api/auth`)

These routes handle custom PBKDF2 student authorization and token-cookie management.

#### 1. `/api/auth/register` (POST)
* **Description**: Register a new student account.
* **Request Payload**: `{ name, email, password }`
* **Internal Logic**:
  1. Validates input format (non-empty fields, secure password length).
  2. Ensures the email does not already exist for active users in the `User` collection.
  3. Invokes `hashPassword` generating a custom random salt and a SHA-512 derived PBKDF2 hash.
  4. Saves the user and calls `setSessionCookie` to log them in automatically.

#### 2. `/api/auth/login` (POST)
* **Description**: Authenticate a student and set cookie.
* **Request Payload**: `{ email, password }`
* **Internal Logic**:
  1. Implements IP-based rate limiting (max 5 tries per minute).
  2. Finds active user by email.
  3. Verifies input password against the stored salt and hash using PBKDF2 derivation.
  4. Sets cookie `auth-token` containing the signed user payload.

#### 3. `/api/auth/logout` (POST)
* **Description**: Destroy user session.
* **Request Payload**: None.
* **Internal Logic**: Clears the `auth-token` cookie by setting its maxAge to 0.

#### 4. `/api/auth/me` (GET)
* **Description**: Fetch current logged-in student details.
* **Headers**: Reads `auth-token` cookie.
* **Internal Logic**: Decodes the session token, verifies the signature, and queries the database for the active user, returning `{ success: true, user }`.

### Group B: Student Portal Core Routes (`app/api/user`)

These routes support the student dashboard, resource discovery, sequential locks, and class unlocks.

#### 5. `/api/user/classes` (GET)
* **Description**: Fetch list of all active classes.
* **Security**: Requires active user session.
* **Internal Logic**: Queries `Class` where `isDeleted: { $ne: true }` and returns the collection.

#### 6. `/api/user/subjects` (GET)
* **Description**: Get active subjects.
* **Params**: `?classId=X`
* **Internal Logic**: Verifies access to the class first, then retrieves subjects for that specific class.

#### 7. `/api/user/tests` (GET)
* **Description**: List tests available under a subject, enforcing sequential locks.
* **Params**: `?classId=X&subjectId=Y`
* **Internal Logic**:
  1. Verifies class access.
  2. Fetches all active tests sorted by creation date.
  3. For each test, determines if it is locked. The backend checks if the student has completed previous tests in that subject sequence. If Test 1 is unattempted, Test 2 is flagged as `locked: true`.

#### 8. `/api/user/mcqs` (GET)
* **Description**: Fetch MCQs for a timed exam. **The correct answers are omitted from the options payload for security.**
* **Params**: `?testId=X`
* **Internal Logic**: Ensures class access, fetches active MCQs for the test, maps them to exclude `correctAnswer`, and returns them.

#### 9. `/api/user/notes` (GET)
* **Description**: Lists preparation PDF notes for a class and subject, indicating whether each note is unlocked or requires a coupon.
* **Params**: `?classId=X&subjectId=Y`
* **Internal Logic**: Queries `Note` collection. Checks if the student has unlocked individual notes via `NoteAccess` records.

#### 10. `/api/user/note-classes` (GET)
* **Description**: Retrieves note classes approved or requested by the user.

#### 11. `/api/user/request-access` (POST)
* **Description**: Submit a payment-based request to unlock a class.
* **Request Payload**: `{ classId, paymentMethod, paymentDetails }`
* **Internal Logic**: Ensures no duplicate pending requests, and creates a `ClassAccess` entry with status `pending`.

#### 12. `/api/user/access-status` (GET)
* **Description**: Check list of class IDs approved for the logged-in student.
* **Internal Logic**: Fetches `ClassAccess` where `userId` matches current user and `status` is `approved`.

#### 13. `/api/user/redeem-coupon` (POST)
* **Description**: Instantly unlock a class by entering a valid coupon code.
* **Request Payload**: `{ classId, code }`
* **Internal Logic**:
  1. Looks up active, unused coupons for `classId`.
  2. Compares raw input `code` using `bcrypt.compare` against hashed DB coupons.
  3. On match, updates coupon status (`isUsed: true`, updates timestamps and user links) and sets `ClassAccess` to `approved`.

#### 14. `/api/user/redeem-note-coupon` (POST)
* **Description**: Unlock all notes under a subject/class via a note coupon.
* **Request Payload**: `{ classId, subjectId, code }`
* **Internal Logic**: Authenticates coupon. On success, writes `NoteAccess` records for all notes inside that class/subject.

#### 15. `/api/user/redeem-note-class-coupon` (POST)
* **Description**: Unlock note class access using a coupon code.

#### 16. `/api/user/submissions` (GET)
* **Description**: Retrieve the student's test history (up to 10 entries).
* **Internal Logic**: Queries `TestSubmission` matching `userId`, populating class/subject/test metadata.

### Group D: Exam Taking & Live Scoring Routes (`app/api/tests`)

#### 32. `/api/tests/[testId]` (GET)
* **Description**: Fetch test information for the student before initiating an exam.
* **Internal Logic**: Checks class approval status. If verified, sends high-level test configurations.

#### 33. `/api/tests/submit` (POST)
* **Description**: Grade student submissions securely (anti-cheat scoring).
* **Payload**: `{ classId, subjectId, testId, startTime, answers: [{ mcqId, selectedOption }] }`
* **Internal Logic**:
  1. Rate limits submissions to maximum 3 per minute.
  2. Queries DB for MCQ answers belonging to the test.
  3. Iterates over student selections, incrementing scores only on matching `correctAnswer` index.
  4. Calculates duration: `Date.now() - startTime`.
  5. Stores the completed attempt inside `TestSubmission` collection.
  6. Returns the submission ID. Correct answers are kept safe and never sent down in plain sight.

## 🔒 5. Essential Security Checkpoints

1. **Server-Grading Only**: The property `correctAnswer` in the `MCQ` schema is marked to be explicitly queried out or is simply not fetched in the student endpoints.
2. **Cheat Prevention Event Block**: Frontend disables mouse right clicks and copy-paste sequences while taking the live exam.
3. **Session Cookies**: Tokens are marked `HttpOnly` and `SameSite: Lax`, defending against XSS and CSRF.
4. **Alphanumeric Bcrypt Coupon Hashes**: If the database is compromised, coupon codes cannot be reverse-engineered since they are hashed using bcrypt.
5. **Rate-Limiting**: Essential endpoints (Login, Submission) protect against brute-force attacks.
